# 22-Criteria OpenAPI Compliance Rubric

Every document generated by DocForge is scored against these 22 criteria.
Each criterion returns YES, NO, or PARTIAL. Score = (YES × 1 + PARTIAL × 0.5) / 22 × 100.

| # | Category | Criterion | Pass condition |
|---|---|---|---|
| Q1 | HTTP semantics | Method matches operation intent | GET for reads, POST/PUT/PATCH/DELETE for writes |
| Q2 | Endpoint definition | Paths clearly defined | All paths present, no placeholders |
| Q3 | Server URL | Non-localhost server present | At least one `servers[].url` that is not localhost or TODO |
| Q4 | Request content type | Content type declared | `requestBody.content` key present and not empty |
| Q5 | Input types | All fields have types | Every `properties` entry has an explicit `type` |
| Q6 | Required status | Required fields marked | `required` array present in schema |
| Q7 | Descriptions | Meaningful descriptions | Each description > 15 characters, not just the field name |
| Q8 | Examples | Example values present | Every parameter has an `example` value |
| Q9 | Defaults | Optional fields have defaults | Fields not in `required` list have `default` set |
| Q10 | Enums | Categorical fields have enums | Flag/boolean/state fields have `enum` list |
| Q11 | Response content type | Response content type declared | `responses[code].content` key present |
| Q12 | Response schema types | Response property types explicit | No bare `type: object` without `properties` |
| Q13 | Response descriptions | Response properties described | Each response property has description > 15 chars |
| Q14 | Response examples | 2+ examples per status | At least 2 named `examples` under each status code |
| Q15 | 2xx coverage | Success schema documented | At least one 2xx response with full schema |
| Q16 | 4xx coverage | Client error documented | At least one 4xx response with schema |
| Q17 | 5xx coverage | Server error documented | At least one 5xx response with schema |
| Q18 | Response enums | Categorical response fields have enums | State/status response fields have `enum` |
| Q19 | Schema consistency | Schemas use `$ref` | All response schemas reference `components/schemas` |
| Q20 | Security scheme | Security defined | `components/securitySchemes` is present and non-empty |
| Q21 | Schema reuse | `$ref` used consistently | No inlined schemas in `paths` — all in `components` |
| Q22 | Version | API version present | `info.version` is present and not empty |

## Score interpretation

| Range | Meaning |
|---|---|
| 80–100% | Publication-ready. Meets professional documentation standards. |
| 60–79% | Good foundation. Review the quick-fix list for specific gaps. |
| 40–59% | Partial. Key sections missing or incomplete. |
| < 40% | Needs significant improvement. Schemas or examples likely missing entirely. |

## Quick-fix hints (generated per failing criterion)

For each NO or PARTIAL result, DocForge generates a specific one-line fix, for example:

- Q3 NO → "Add a `servers:` entry with the service's actual base URL"
- Q8 PARTIAL → "Add `example:` to parameters: mobile, sms_content"
- Q14 NO → "Add 2 named examples under `responses.200.content.application/json.examples`"
- Q19 NO → "Move inline response schema to `components/schemas/ResponseName` and use `$ref`"
